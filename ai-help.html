AI Help

How to ask for the result you want, what context the AI actually sees, and why a loaded machine profile usually produces more useful GCOM on the first pass.

Current context

Controller guidance only

This page mirrors the same context flow used by the AI request in the main app.

Fast rule

Describe the motion pattern, not just the idea.

The AI improves when you state the exact loop math, the start and end coordinates, whether a move should be G0 or G1, and what the output must not do.

If a previous answer scaled values, inserted ROUND(), or chose the wrong geometry, say not to do that in the next prompt.

Where AI gets context

The AI sees more than the latest chat message.

Current script

Your script source, metadata fields, and variable defaults are attached so the AI can modify or continue the program you already have.

Validation state

Current syntax errors, warnings, and the validation summary are included, so the AI can explain or repair the exact issue the editor is showing.

Preview selection

If you select commands or segments in Preview, the AI also gets those commands, motion segments, line numbers, and geometry summaries. This is the fastest way to focus it on a wrong move.

Terminal output

If you enable the terminal toggle, or the app saw a recent script error, recent terminal lines are attached automatically.

Controller profile

The AI always gets the active controller guidance: controller flavor, preferred style, avoid list, acknowledgement behavior, and boilerplate GCOM.

Machine profile

If a machine profile is loaded, or a ;PROFILE: header points to one, the AI also gets machine notes, exact command examples, snippets, presets, and output-scaling conventions.

Why prompts drift

Common missing details

  • You describe the visual result, but not the actual move order.
  • You name variables, but not how each variable should compound through the loop.
  • You do not say whether values should be used literally or transformed by scaling or rounding.
  • You do not specify whether the next line begins with a rapid reposition or a cut.
  • You do not say what extra moves are forbidden.
What to include

High-yield prompt details

  • Name the required variables exactly.
  • Spell out the loop in plain language: for each i from 0 to number_of_lines - 1.
  • Give the coordinate formulas for start and end points.
  • Name the command types explicitly: G0 to reposition, G1 to cut.
  • State the prohibitions: do not scale, do not round, do not add extra return segments.
Prompt refinement example

Your flared-cuts conversation needed the geometry to become explicit.

Earlier prompt
I'd like to generate a series of horizontal cuts with variables controlling the spacing at each end so one end can optionally be flared out. power at 30%, speed 3000mm/min.
  • It did not define the exact G0 and G1 sequence.
  • It did not define number_of_lines.
  • It did not say whether power, flare_start, and flare_end should be used literally.
  • It did not say whether the flare should appear as horizontal spacing, diagonal endpoints, or a return pass.
Prompt that would have got you there much earlier
I'd like to generate a series of horizontal cuts with variables controlling the spacing at each end so one end can optionally be flared out. The cuts should start at X0 Y0 and end at X100 Y0, then move rapidly to X0 Y1 and cut to X100 Y2, and so on. The first cut should start at X0 Y0 and end at X100 Y0, the second cut should start at X0 Y1 and end at X100 Y2, and so on. Use a number_of_lines variable to control the number of lines. Power should be 30% (300) and speed should be 3000 mm/min. Use ROUND to round the coordinates and feed rates to 4 decimal places. Use M4 for dynamic power scaling.

That version removes the biggest sources of drift: it fixes the exact variable set, the line length variable, the use of M4 and M5, the loop math, the retained ROUND formatting, and the ban on extra return segments.

Say this early

Use these variable names exactly: power, speed, flare_start, flare_end, number_of_lines, length_of_lines.

Say this when needed

State explicitly whether emitted values should keep ROUND(..., 4) formatting or stay completely unrounded.

Say this for geometry

Describe each loop iteration as coordinate formulas, not only as words like flared, offset, or stepped.

Machine profiles help

Without a machine profile, the AI only has controller-level guidance.

No machine profile loaded
  • Controller flavor label
  • Preferred style and avoid guidance
  • Recommended boilerplate GCOM
  • General acknowledgement behavior
Machine profile loaded
  • Machine name and notes
  • Exact command examples such as job_start, tool_on_sequence, rapid_move, and cut_move
  • Saved snippets and presets
  • Output-scaling conventions like what S means and the intended range

These are injected as plain GCOM references, which is why a good machine profile often improves first-pass results more than wording tweaks alone.

Best workflow

Use the AI panel like a constrained editor.

  1. Load the right controller profile and, if you have one, the right machine profile.
  2. Select the wrong commands or segments in Preview before asking for a fix.
  3. Describe the exact coordinate formulas or loop behavior you expect.
  4. Name any hard constraints: no scaling, no rounding, no extra segments, preserve the current structure.
  5. If the result is close, ask for a targeted correction instead of regenerating the entire script.
Target structure

The working pattern you can load directly

; TITLE: Flared Horizontal Cuts
; VERSION: 1
; AUTHOR: GitHub Copilot
; DESCRIPTION: Cut a series of horizontal lines with variable spacing at each end.
; VAR flare_end=0.15
; VAR flare_start=0.03
; VAR number_of_lines=10
; VAR power=300
; VAR speed=3000
; VAR length_of_lines=350

10 LET power_s = {power}
20 LET speed_f = {speed}
30 SEND "M4" REQUIRE_OK
40 SEND "G0 X0 Y0 F" & speed_f & " S" & power_s REQUIRE_OK
50 FOR i = 0 TO {number_of_lines} - 1 STEP 1
60 LET y_start = i * {flare_start}
70 LET y_end = i * {flare_end}
80 SEND "G0 X0 Y" & ROUND(y_start, 4) & " F" & ROUND(speed_f, 4) & " S" & ROUND(power_s, 4) REQUIRE_OK
90 SEND "G1 X" & {length_of_lines} & " Y" & ROUND(y_end + {flare_end}, 4) & " F" & ROUND(speed_f, 4) & " S" & ROUND(power_s, 4) REQUIRE_OK
100 NEXT i
110 SEND "M5" REQUIRE_OK
120 END

Load sends this full metadata-bearing example back to the main app using the same help-window message path as Basic help, so the title, description, and VAR defaults load with the script.
